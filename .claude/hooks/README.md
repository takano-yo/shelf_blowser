# .claude/hooks — フック

Claude Code が `.claude/settings.json` に基づいて実行するシェルフック。
（フックは**シェルを実行する仕組み**で、Claude の skill は呼べない。new-branch と
同等の挙動を git コマンドで再現している。）

## post-merge-rebranch.sh（UserPromptSubmit フック）

**一つのチャットで開発している間、PR が main にマージされるたびに、自動で
新しいブランチへ移行し、マージ済みの旧ブランチを片付ける。**

ユーザーがプロンプトを送信した時点（UserPromptSubmit）= **Claude がそのプロンプトを
処理する「前」**に発火し、次の条件を**すべて**満たすときだけ動く。

1. 既定ブランチ（`main` / `master`）上でも detached HEAD でもない。
2. **active（main 以外のローカル）ブランチが 1 つだけ**である。
   - 複数ある場合は誤削除を避けるため**何もしない**（ユーザーに委ねる）。
3. 現在のブランチが `origin/main` に**マージ済み**である（かつ main と同一の
   作りたてブランチではない）。
4. 未コミットの変更が無い（あるときは自動切替を見送り、警告のみ）。

満たしたときの動作:

1. `git fetch origin main` で最新を取得。
2. 最新 `main` から新しいブランチ `claude/work-<日時>` を作成して移行。
3. マージ済みの旧ブランチを `git branch -d` で削除。
4. `systemMessage` で「削除した旧ブランチ名 / 新ブランチ名」を通知。

### なぜ UserPromptSubmit か（Stop ではなく）

当初は `Stop`（Claude が応答を返し終えた後）で発火させていたが、次の時間差があった。

```
[merge] → [次の指示] → [Claude がファイル変更] → [Stop: リブランチ]
                ↑──── この間、変更が古いマージ済みブランチに乗ってしまう ────↑
```

さらに、その間に**コミット**すると旧ブランチが main の祖先でなくなり、マージ判定が
外れて**リブランチされない**問題もあった。

`UserPromptSubmit` は**プロンプトを処理する前**に発火するため、

```
[merge] → [次の指示を送信] → [UserPromptSubmit: 先にリブランチ] → [新ブランチで変更]
```

となり、ファイル変更が常に新しいブランチに乗る。プロンプト送信時点は作業ツリーが
クリーンなことが多く、切替時の巻き込みリスクも小さい。

### 設計方針

- **fail-open**: git 操作やネットワークが失敗しても、フックは安全に終了し
  プロンプト処理を妨げない（ブロックしない）。
- **安全側に倒す**: active ブランチが 1 つ・マージ済み・未コミット変更なしの
  ときだけ自動化する。判断がつかない状況では何もしない。
- 新ブランチ名は日時ベースの自動生成。意味のある名前にしたいときは作成後に
  `git branch -m <新名>` でリネームする。

### 制約

- マージ済み判定は `git merge-base --is-ancestor`（**マージコミット方式**）を前提。
  GitHub の **squash / rebase マージ**では旧ブランチの tip が main の祖先に
  ならないため、この方法では検知できない。本リポジトリの PR は「Merge pull
  request」（マージコミット）で運用しているため問題ない。
- `UserPromptSubmit` はプロンプト送信ごとに発火するため、毎回 `git fetch
  origin main` が走る（条件を満たさない大半は fetch のみで終了する）。
- merge 後にユーザーが何も送らずアイドルの間はリブランチされない（次の送信時に行う）。

### 無効化

一時的に止めたいときは `.claude/settings.json` の `UserPromptSubmit` フック項目を
削除するか、スクリプトを実行権限から外す（`chmod -x`）。
