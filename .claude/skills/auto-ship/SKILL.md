# auto-ship — 軽微な変更を自動でコード作成 → PR → main マージまで

安全性の確認が不要な**軽微な変更**に限り、Claude が
**コード作成 → コミット → PR 作成 → 整合性確認 → main へのマージ**までを一気通貫で行う。

ただし、**少しでも注意が必要な兆候があれば自動マージは行わない。**
その場合は PR 作成までで止め、何が懸念なのかをユーザーに明示して判断を仰ぐ。

> **大原則**: 自動マージは「やり直しが容易で、影響範囲が局所的で、危険な操作を一切
> 含まない変更」にだけ許される。判断に迷ったら自動マージしない。安全側に倒す。

## 引数

```
/auto-ship <やりたいこと> [ブランチ名]
```

- `<やりたいこと>`: 実施する軽微な変更の内容（必須）。
- `[ブランチ名]`: 省略時は変更内容から短いケバブケース名を生成する
  （例: `fix/typo-readme`, `chore/update-link`）。

## 自動マージしてよい「軽微な変更」の条件（すべて満たすこと）

以下を **すべて** 満たす場合のみ自動マージまで進める。

- 変更が**局所的**（少数ファイル・小さな差分）で、ロジックの根幹を変えない。
- typo 修正・ドキュメント/コメント更新・文言調整・軽微なバグ修正・テスト追加・
  フォーマット整形・小さな設定値変更など、**振る舞いへの影響が読み切れる**もの。
- **既存テスト/ビルドが通り**、参照崩れ（壊れたリンク・import・パス）が無い。
- 後述の「注意が必要な項目」に**一切該当しない**。

## 自動マージを止めてユーザーに注意喚起すべき項目

次のいずれかに**少しでも該当・関係する**場合は、**自動マージしない**。
PR 作成（または変更提示）までで止め、該当箇所と理由を具体的に伝えて確認を求める。

- **外部への侵入・不正アクセス**につながりうるコード（スキャン、認証回避、
  権限昇格、未許可のリモートアクセス等）。
- **大量ファイルの削除・破壊的操作**（広範な `rm`、ディレクトリ一括削除、
  履歴の破壊的書き換え、`--force` での上書き、データ消去等）。
- **攻撃的・悪用可能なコマンド/スクリプトの作成**（DoS、マスターゲティング、
  検知回避、マルウェア的挙動等）。
- **秘密情報・認証情報**の追加/変更/露出（トークン、鍵、パスワード、`.env` 等）。
- **依存関係の追加・更新**やサプライチェーンに関わる変更（lockfile、新規パッケージ）。
- **CI/CD・ワークフロー・権限設定**の変更（`.github/workflows/`、権限スコープ等）。
- **広範囲・大規模な差分**、多数ファイルにまたがる変更、公開 API/スキーマ/DB
  マイグレーションの変更。
- ユーザーが**レビューしたい意向を示している**、または影響範囲が読み切れないもの。

> 該当時の振る舞い: マージは行わず、「自動マージを保留した理由」「該当箇所」
> 「想定される影響」「次の選択肢（手動マージ/修正/中止）」を提示する。
> 勝手に進めない。

## 手順

1. **トリガー判定**
   依頼内容を読み、上の「注意が必要な項目」に該当しないか最初に判断する。
   該当する場合は以降の自動マージは行わず、PR 作成までに切り替える旨を先に伝える。

2. **ブランチを切る**
   最新 `main` を起点に作業ブランチを作成する（`new-branch` スキルの手順に準拠）。
   未コミットの変更があれば勝手に破棄せず、扱いをユーザーに確認する。

3. **変更を実装してコミット**
   依頼内容を実装する。差分が「軽微の条件」を逸脱して大きくなった場合は、
   いったん止めて自動マージの可否を再判定する。
   コミットメッセージは変更内容を端的に記す。

4. **整合性を確認（マージ前の必須ゲート）**
   マージ前に次を確認し、**一つでも失敗・不明なら自動マージしない**。
   - リポジトリにテスト/ビルド/リンタがあれば実行し、**グリーン**を確認する。
   - 参照崩れ（壊れたリンク・import・相対パス）が無いことを確認する。
   - 変更が依頼の範囲内に収まり、意図しない副作用が無いことを差分で確認する。
   - 「注意が必要な項目」に該当しないことを最終確認する。

5. **PR を作成**
   ブランチをプッシュし、PR を作成する（リポジトリに PR テンプレートがあれば踏襲）。
   PR 本文に変更概要・整合性確認の結果（実行したテスト/ビルド等）を記載する。

6. **CI を確認**
   PR の CI/チェックがあれば**成功を待つ**。失敗・保留中は自動マージしない。

7. **マージ判定と実行**
   - **軽微の条件をすべて満たし、注意項目に非該当、整合性 OK、CI 成功**のときのみ、
     `main` へマージする。マージ後はその旨と PR リンクを報告する。
   - いずれか満たさない場合は**マージせず**、保留理由・該当箇所・次の選択肢を提示して
     ユーザーの判断を仰ぐ。

## 注意

- 不可逆・外向きの操作（マージ、push、削除）は安全側に倒す。少しでも疑わしければ
  自動マージしない。
- このスキルは「軽微で安全な変更の自動反映」を目的とする。判断がグレーなものを
  無理に軽微扱いしない。グレーは必ずユーザー確認に回す。
- ユーザーが途中で停止・保留を求めたら直ちに従い、それ以上マージを進めない。
