# サイト内検索コーパス — PoC・実測結果（実装手順 1）

> [docs/site-search.md](site-search.md) 実装手順 1「PoC・実測」の成果物。
> コーパス（方式①）を試作し、**生成時間・転送量（gzip 実測）・走査時間・
> 書誌解決のファイル数分布**を計測して、**取得上限 K とコーパス形式**を確定する。
> 方式②（バイグラム転置索引）との比較判断もここで行う。
>
> 計測スクリプト: [`build/poc/search_poc.py`](../build/poc/search_poc.py)

## 計測方法

- 対象データ: 現収録の `site/data/ndc/`（全 1,110 分類・生 368MB）。
- コーパス 1 行 = **類内ユニーク書誌**（`ncid` で重複排除）。内容は
  **正規化済み検索文字列＋書誌本体への参照**（収録分類記号＋その分類ファイル内の
  行位置）のみ。書誌本体は `site/data/ndc/<記号>.json` から解決する（二重保存しない）。
- 参照する分類記号は **その書誌を含む最も浅い（記号が短い）分類ファイル**を選ぶ
  （所蔵館数上位ほど 1 桁ファイル＝類全体の上位 1,000 に入るため、上位ヒットを
  少数の浅いファイルへ寄せる狙い）。
- 正規化は本書 [照合仕様](site-search.md#照合仕様収録データ検索) の 4 段（NFKC →
  小文字化 → カタカナ→ひらがな → 小書き仮名→並字）を PoC 側に内蔵して適用。
- 実行環境: 本リポジトリの実行コンテナ（Python 3・単一プロセス）。走査時間は
  デスクトップ級 CPU の目安。モバイル実機は 2〜3 倍を見込む。

## 結果 2: 走査時間（`includes()` 部分一致 AND）

代表クエリ（`日本` / `歴史` / `研究` / `東京 大学` / `python` / `コンピュータ` /
`経済学`）で **1 類 8〜9 万件の走査は 30〜77ms**。メモリ上の正規化済み文字列同士の
`includes()` で、クライアント負荷は問題にならない（モバイルでも体感即時の範囲）。

## 結果 3: 書誌本体の解決に要する転送量（**K 確定の核心**）

上位 1,000 件のヒットを描画するには、参照先の分類ファイル（各 ≒ 1,000 件・生 ≒
400KB）を取得する必要がある。**ヒット件数の多いファイルから順に**取得したときの
カバー件数と累積転送（生）を計測した。

| クエリ傾向 | 例 | 解決ファイル数 | K=20（生） | K=30（生） | 全件（生） |
|---|---|---|---|---|---|
| 高所蔵に集中 | `日本`(類3, 13,843件) | 19 | 1000件 / 7.9MB | 1000件 / 7.9MB | 7.9MB |
| 集中 | `研究`(類6, 6,463件) | 42 | 904件 / 8.1MB | 977件 / 11.9MB | 16.6MB |
| **広く分散** | `歴史`(類5, 1,023件) | **94** | 604件 / 7.7MB | 733件 / 11.7MB | **36.8MB** |
| **広く分散** | `東京 大学`(類5, 891件) | **90** | 486件 / 8.1MB | 627件 / 12.0MB | **35.7MB** |
| 少数語 | `python`(類0, 45件) | 5 | 全45件 / 2.2MB | — | 2.2MB |

**所見**:

- **参照＋分類ファイル取得方式は転送が重い**。分類ファイルは 1,000 件束の ≒ 400KB
  なので、そこから数件を取り出すために 1 ファイル丸ごと取得することになる。
  上位 1,000 件が多数の細目へ散る「広く分散」クエリ（`歴史`・`東京 大学` 等）では
  **全件表示に 30〜37MB**（要件定義 問題点 #2 の「最悪 111 ファイル ≒ 生 36MB」を実測で確認）。
- **K=20（≒ 8MB）**: `日本`・`研究` 等の集中クエリはほぼ 1,000 件を満たすが、
  分散クエリは **250〜700 件**しか表示できない。
- **K=30（≒ 12MB）**: 集中クエリは 977〜1000 件、分散クエリでも 600〜900 件へ改善。
- ファイル数とバイトはほぼ比例（≒ 400KB/ファイル）。**K はファイル数と
  バイト予算のどちらで表現しても実質同じ**。

## 結果 4: コーパス形式の比較（類 0）

| 形式 | 生 | gzip |
|---|---|---|
| array-of-objects `[{"s","f","i"}, …]` | 8.08MB | 2.56MB |
| array-of-arrays `[["s","f",i], …]` | 7.55MB | 2.52MB |
| **列指向 `{"s":[…],"f":[…],"i":[…]}`** | **7.46MB** | **2.38MB** |

**列指向が最小**（キー重複が消える）かつ、走査時は正規化文字列配列 `s[]` だけを
読み、ヒットした行だけ `f[]`/`i[]` を参照すればよく**走査効率も最良**。

## 結果 5: 正規化の確認（原文 → 正規化後）

| 原文 | 正規化後 | 効いた規則 |
|---|---|---|
| `Python プログラミング` | `python ぷろぐらみんぐ` | 小文字化・カナ→かな |
| `ＣＯＭＰＵＴＥＲ` | `computer` | NFKC 全角→半角・小文字化 |
| `ｶﾀｶﾅ`（半角） | `かたかな` | NFKC 半角カナ→全角・カナ→かな |
| `ヴァイオリン` | `ゔあいおりん` | ヴ→ゔ・小書き ァ→あ |
| `キャッシュ` | `きやつしゆ` | 小書き ャ→や・ッ→つ・ュ→ゆ |
| `①巻` | `1巻` | NFKC 互換分解（許容） |
| `コンピューター` | `こんぴゆーたー` | 長音「ー」はそのまま（`コンピュータ`＝`こんぴゆた` とは非一致） |

規則どおりに動作。長音の除去・濁点の統一は行わない（`コンピュータ`≠`コンピューター`、
`は`≠`ば`）ため、それらは意図どおり別語として扱われる。

## 結果 6: 方式②（バイグラム転置索引）との比較

| 方式 | 1 類の索引規模（概算） | 検索の転送 | 実装 |
|---|---|---|---|
| ① コーパス走査（本 PoC） | 生 8〜16MB（gzip 2.5〜5.2MB） | 初回に類コーパス（gzip 2.5〜5.2MB）、以後 0 | 単純・偽陽性なし |
| ② バイグラム転置索引 | **postings で 11〜24MB** | 検索語のバイグラム分（数百 KB／検索） | 複雑・分割設計・偽陽性検証が別途必要 |

- 方式②の postings 索引は**コーパスより大きい**（類 9 で概算 23.6MB）。1 検索あたりの
  転送は小さいが、**索引の分割配信・偽陽性の除去・1 文字語の扱い**が別途必要で複雑。
- 本件の主眼（サーバ未稼働の Pages で、類コーパスを 1 回読めば以後即時）には
  **方式①で十分**。方式②はモバイル初回転送が実測で problem になった場合の代替に留める。

## 確定事項（手順 2 以降の前提）

1. **方式は ① コーパス走査を採用**（方式②は不採用。将来の代替として記録）。
2. **コーパス形式は列指向 JSON**: 類ごとに `corpus-<類>.json` を
   `{"s":[正規化済み検索文字列…], "f":[収録分類記号…], "i":[行位置…]}` で出力する。
   並びは類内の所蔵館数降順・同値 ncid 昇順（ランキング順）。走査は `s[]` のみを読む。
   - 参考: 参照 `i`（行位置）は「そのデプロイの `site/data/ndc/<f>.json` の並び」に
     依存する。コーパスも同じ ndc から同時生成するため整合する（手順 5）。
3. **書誌解決の取得上限 K = 30 ファイル**（≒ 生 12MB）を既定とする。
   **ヒット件数の多い分類ファイルから順に**取得し、K に達したら以降は打ち切って
   棚上部に「上位 m 件を表示」を明示する（要件定義 UI 仕様・問題点 #2）。
   - 集中クエリは K=30 でほぼ 1,000 件を満たす。広く分散するクエリ（`歴史` 等）は
     600〜900 件で頭打ちになるが、これは**要件定義が事前に許容した挙動**（#2）。
4. **走査・正規化・コーパス規模は要件定義の見積りどおり実装可能**であることを確認した。

### 参考（将来の改善余地・本 PoC のスコープ外）

- 分散クエリで「全 1,000 件・低転送」を両立するには、コーパスへ表示本体を同梱して
  二次取得を無くす案（＝1 類 1 ファイルで完結）がある。ただし**書誌本体の二重保存
  （+ 約 112MB）**となり、要件定義の確定事項（二重保存しない）と相反するため、
  **採用するならユーザー判断が必要**。本 PoC では既定方針（参照＋K 打ち切り）を採る。
