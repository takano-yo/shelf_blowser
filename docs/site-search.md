## 背景（現状の問題）

サーバー未稼働（GitHub Pages のみ）のとき、検索は次のように縮退している。

| 入口 | 現状の挙動 |
|---|---|
| NDC 棚（`?ndc=<記号>`）内の検索 | 読み込み済みの棚データ（**最大 1,000 件**）のクライアント側絞り込みに縮退 |
| スタートページからの直接検索（`?q=<語>`） | `/api/search` が 404 → **既定棚（日本近代文学）へフォールバック**し検索できない |

一方、収録データは全 1,110 分類 × 最大 1,000 件（延べ 936,548 行・ユニーク 774,917 件・
生 367MB）あり、これを活かせていない。

## 決定事項（2026-07-15・ユーザー確定）

1. **検索範囲は NDC 1 桁（類・0〜9）ごと**とする。収録データ検索は常に 1 つの類を
   対象にする（全 1,110 分類の横断検索はしない）。
2. **スタートページ・本棚ページの双方に検索モード切替**を置く:
   - **収録データ検索（NDC 類内検索)** … 事前取得済みデータから検索。**サーバー不要**。
   - **CiNii API 検索** … `/api/search` による直接検索。**サーバー稼働時限定（現状維持）**。
   - 可能なら（本書では実装スコープ内とする）**サーバー稼働の有無を判別し、
     未稼働時は API 検索の選択肢をグレーアウト**（選択不可＋理由表示）する。
3. **検索結果は最大 1,000 件・所蔵館数降順（既定）**。既存の並べ替え
   （出版年 昇順/降順）は検索結果に対しても維持する。
4. **1 文字クエリは検索不可**とし、「2 文字以上で入力してください」と促す。
5. **正規化はひらがな/カタカナ・大文字/小文字（ひらがな・カタカナの小書きを含む）・
   全角/半角のみ**行う（詳細は「照合仕様」）。
6. **検索コーパス（索引）は GitHub Pages デプロイ時に生成**する（リポジトリへは
   コミットしない）。

## 検索モードの仕様

| モード | 対象データ | 範囲 | サーバー |
|---|---|---|---|
| 収録データ検索（既定） | `site/data/ndc/` の事前取得データ | 選択した **類（1 桁）** の収録全件 | 不要 |
| CiNii API 検索 | CiNii Books（ライブ）／ローカル代役 | 全体（NDC 棚では `clas=<記号>*` の分類内。**現状の挙動を維持**） | 必須 |

- **既定モードは収録データ検索**。従来の「サーバー優先 → 失敗時に縮退」から
  「**静的検索が既定・API 検索は明示切替**」へ挙動を変更する（Pages 公開が主経路のため）。
- **サーバー稼働判定**: サーバーに軽量エンドポイント **`/api/ping`**（`{"ok": true}` を
  返すだけ）を新設し、ページ読込時に 1 回 fetch して判定する。失敗（404・ネットワーク
  エラー・タイムアウト）なら API 検索の選択肢をグレーアウトし、「検索サーバー未稼働」の
  理由を表示する。API 検索の実行がエラーになった場合も同様にグレーアウトへ落とす。

## 照合仕様（収録データ検索）

- **検索対象フィールドは現行と同じ**: タイトル・著者（`creatorRaw`・`creators`）・
  出版社・シリーズ名（`bookHaystack` / server の `_book_haystack` と同一の対象）。
  タイトルのみ等への限定はしない。
- **空白区切りの AND・部分一致**（現行と同じ意味論）。
- **各検索語は 2 文字以上**。1 文字の語が含まれる場合は検索を実行せず
  「2 文字以上で入力してください」と表示する（両モード共通）。
- **正規化**: コーパス構築側（Python）とクエリ側（JS）の双方へ、次の順で同一規則を
  適用して照合する（**表示は原文のまま**）:
  1. **Unicode NFKC 正規化** … 全角英数記号→半角、半角カナ→全角カナ等の全角/半角統一。
     （副作用として ㈱→(株)・①→1 等の互換分解も行われる。許容する）
  2. **英字の小文字化**（A→a）
  3. **カタカナ→ひらがな**（ヴ→ゔ を含む。長音「ー」はそのまま）
  4. **小書き仮名→並字**（ぁぃぅぇぉっゃゅょゎゕゖ → あいうえおつやゆよわかけ）
  - 濁点/半濁点の統一（は≠ば）・長音の除去は**行わない**。
  - Python と JS の実装一致は**共有テストベクタ**（正規化前→後の対を収めた JSON を
    リポジトリに置き、双方のテストから読む）で担保する。
- **ランキング**: 類内の所蔵館数（`ownerCount`）降順・同値は ncid 昇順
  （既存の整列規則と同一）。**上位 1,000 件で打ち切り**、超過時は
  「全 n 件中 上位 1,000 件を表示」と棚上部に明示する。
- 並べ替え（出版年 昇順/降順）・タブ・シリーズまとめは、結果集合（≤1,000 件）に
  対して既存実装のまま動作する（追加実装なし）。

## UI 仕様

### スタートページ（`site/index.html`）

- 検索窓に**モード切替**（収録データ検索 / CiNii API 検索）を併設する。
- **収録データ検索モードでは類セレクタ（0〜9・分類名つき）**を表示し、
  類を選んで送信 → `shelf.html?ndc=<類>&q=<語>` へ遷移する。
- API 検索モードでは現行どおり `shelf.html?q=<語>&mode=api` へ遷移する
  （検索の実行は本棚ページ側。スタートページは API を叩かない方針は維持。
  ただし `/api/ping` による稼働判定のみ行い、未稼働時は API モードをグレーアウト）。

### 本棚ページ（`site/shelf.html`）

- 検索バーに**モード切替**を置く。NDC 棚（`?ndc=<記号>`）表示中の収録データ検索は、
  **記号の 1 桁目の類**を検索範囲とする（例: `?ndc=913` の棚からの検索は類 9 全体。
  類セレクタは棚の類を初期選択とする）。
- 既定棚（クエリなし・日本近代文学）からの収録データ検索は類セレクタで類を選ぶ。
- 検索結果の棚上部に**検索範囲の明示**を出す:
  「NDC n 類の収録データ（各分類 所蔵館数上位 1,000 件）から検索」＋件数＋出典。
  収録データはスナップショットであり CiNii 全体（約 1,870 万件）ではないことが
  伝わる文言にする。
- 空送信の挙動は現行を維持（NDC 棚なら分類全体の棚へ、通常は既定データへ戻る）。
- **検索リクエストの競合対策（AbortController・ロードマップ P4 #13）を本件へ統合して
  実装する**（複数ファイル fetch が走るため必須になる）。

## URL 仕様（状態同期の拡張）

| URL | 意味 |
|---|---|
| `?ndc=<記号>&q=<語>` | **収録データ検索**（範囲は記号 1 桁目の類）。空送信で `?ndc=<記号>` へ戻る |
| `?ndc=<記号>&q=<語>&mode=api` | API の分類内検索（`clas=<記号>*`・現状の挙動） |
| `?q=<語>&mode=api` | API の全体検索 |
| `?q=<語>`（mode なし） | **後方互換**: API の全体検索として扱う。サーバー未稼働時は従来の 404→既定棚ではなく、**既定棚＋「収録データ検索へ切り替えて類を選ぶ」案内**を表示する |

- `mode` の不正値は無視して既定へ落とす（既存の `tab`/`sort` と同じ方針)。
- タブ・並べ替えの URL 同期（`tab`/`sort`）は現行のまま。

## データ設計（検索コーパス）

```
site/data/search/               # ← Pages デプロイ時に生成（Git 管理外・.gitignore）
├── manifest.json               # 生成日時・正規化規則バージョン・類ごとのファイル一覧と件数
└── corpus-<類>.json            # 類ごとの検索コーパス（0〜9 の 10 本）
```

- **コーパス 1 行 = 類内のユニーク書誌 1 件**（ncid で重複排除。同一書誌が
  類目/綱目/細目の複数ファイルに収録される階層重複・複数分類重複を除く）。
  内容は「**正規化済み検索文字列**＋**書誌本体への参照**（収録ファイル＝分類記号と
  行位置）」のみで、書誌本体は既存の `site/data/ndc/<記号>.json` から解決する
  （データの二重保存はしない）。並びは類内の所蔵館数降順（＝ランキング順）。
- **検索の流れ**: ① 対象類のコーパスを初回検索時に fetch（以後セッション内で再利用)
  → ② メモリ上で部分一致走査（正規化済み同士の `includes()`・AND）
  → ③ 上位 1,000 件の参照から必要な分類ファイルを fetch して書誌本体を解決
  → ④ 既存の `setBooks()` へ渡して棚を描画。
- **書誌本体の取得上限**: ③ で取得する分類ファイル数に上限 K を設ける
  （目安 20〜30・PoC で確定）。上限に達したら以降のヒットは打ち切り、
  「上位 m 件を表示」と明示する。
- 形式は**テキストベース（JSON）**とする（GitHub Pages の CDN 圧縮
  〈gzip/brotli〉が効く形式にする。バイナリだと圧縮対象外になりうる）。

### 規模の実測（2026-07-15・現収録データで実測）

| 類 | 収録行数 | ユニーク件数 | 検索文字列量（生） |
|---|---|---|---|
| 0 | 56,070 | 43,975 | 6.7MB |
| 1 | 95,398 | 79,474 | 11.0MB |
| 2 | 84,916 | 71,855 | 10.6MB |
| 3 | 109,106 | 91,988 | 12.5MB |
| 4 | 94,775 | 79,474 | 10.8MB |
| 5 | 106,611 | 92,442 | 12.7MB |
| 6 | 99,934 | 86,709 | 12.1MB |
| 7 | 101,433 | 87,901 | 11.5MB |
| 8 | 88,608 | 75,982 | 11.4MB |
| 9 | 99,697 | 86,541 | 13.1MB |

- コーパスは **1 類あたり生 7〜13MB・gzip 転送でおよそ 1/3 以下（2〜4MB 想定）**。
  初回検索時に 1 回だけ読み、以後は同じ類の再検索が即時になる。
- 全類合計は生 約 112MB。Pages のサイト上限（1GB）に対し、
  既存データ 367MB ＋ コーパス 112MB ≒ 479MB で収まる。
- 10 万件規模の `includes()` 走査は数十 ms オーダーで、クライアント負荷は問題ない。

### 方式の位置づけ（コーパス走査を第一候補とする理由）

| 方式 | 1 検索の転送 | 実装 | 備考 |
|---|---|---|---|
| **① コーパス走査（採用・第一候補）** | 初回 2〜4MB（類ごと・以後 0） | 単純 | 完全一致・偽陽性なし。正規化も走査側だけで完結 |
| ② バイグラム転置索引（代替） | 数百 KB／検索 | 複雑 | 分割設計・偽陽性の検証・1 文字語の扱いが別途必要。①の初回転送が実測で問題になった場合の代替 |

いずれもデプロイ時生成のためリポジトリは増えない。①で開始し、PoC の実測
（モバイル回線での初回転送の体感）で問題があれば②へ切り替える。

## 問題点と対処（実装前の確認事項）

| # | 問題 | 対処 |
|---|---|---|
| 1 | **正規化の二重実装**（Python＝コーパス構築 / JS＝クエリ・走査）の不一致リスク | 変換規則を本書で固定し、**共有テストベクタ**を `core` に置いて双方のテストで検証する。manifest に正規化規則バージョンを記録し、不一致時は再生成 |
| 2 | **書誌本体の解決に要するファイル数**。汎用語では上位 1,000 件が類内の多数の分類ファイルへ散らばる（最悪 111 ファイル ≒ 生 36MB） | 取得上限 K（目安 20〜30）で打ち切り＋「上位 m 件を表示」の明示。**K と打ち切り発生頻度は PoC で実測して確定** |
| 3 | **モードごとの結果差**。収録データ検索はスナップショット（各分類 所蔵館数上位 1,000 件）が対象で、CiNii ライブ検索と件数・内容が異なる。API 検索は正規化なし（現状維持）のため照合もわずかに異なる | 棚上部の検索範囲表示で対象データを明示（UI 仕様）。挙動差は仕様として本書に記録 |
| 4 | **`?q=` のみの後方互換**。旧リンクはサーバー前提の意味を持つ | URL 仕様のとおり API 全体検索として扱い、未稼働時は案内つきで既定棚へ（従来の無言フォールバックをやめる） |
| 5 | **CI/CD の変更**（`pages.yml` へのコーパス生成ステップ追加）はユーザーレビューが必要 | 実装手順 5 を独立の PR とし、自動マージ対象にしない（auto-ship の注意項目にも該当） |
| 6 | **ローカル開発**では Pages デプロイ時生成物が無い | `python build/build.py --search-index` の手動実行手順を README に記載。`site/data/search/` は `.gitignore` に追加 |
| 7 | **収録データ更新との整合**。`site/data/ndc/` の再取得後はコーパスも古くなる | デプロイ時生成のため**コミットと同時に自動で追随**する（デプロイごとに再生成）。ランブックへの追記のみ行う |

## 受け入れ条件（この拡張の完了定義）

1. GitHub Pages（サーバー未稼働）で、スタートページ・本棚ページのどちらからも
   **類（1 桁）単位の収録データ検索**ができ、最大 1,000 件・所蔵館数降順で表示される。
2. 両ページに検索モード切替があり、**サーバー未稼働時は API 検索がグレーアウト**
   （選択不可＋理由表示）される。稼働時の API 検索は現状の挙動を維持する。
3. 1 文字の検索語では検索が実行されず、「2 文字以上で入力してください」と案内される。
4. 正規化（全角/半角・英字大小・かな種・小書き）が Python / JS で一致し、
   共有テストベクタで検証されている。
5. 既存 URL（`?q=` のみ・`?ndc=<記号>&q=<語>`）が後方互換の意味で動作する。
6. 検索コーパスは Pages デプロイ時に生成され、リポジトリにコミットされない。
   ローカルでは `build.py --search-index` で生成できる。
7. 検索結果でもタブ・並べ替え（出版年 昇順/降順）・シリーズまとめが従来どおり動く。
