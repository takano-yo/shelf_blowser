## 全体像

| ページ | 役割 | ファイル（実装方針） |
|---|---|---|
| **スタートページ** | 入口。検索窓と NDC 分類ボタンで**初期条件**（検索語 or NDC 分類）を指定する | `site/index.html`（新規） |
| **トップページ（本棚ページ）** | 現行の本棚 UI。初期条件に応じた棚を表示し、検索時は棚を再表示する（現行どおり） | `site/shelf.html`（現行 `index.html` を移設） |

- サイトの公開ルート（`/`）は**スタートページ**になる。
- 本棚ページは **URL クエリで初期条件を受け取る**:
  - `shelf.html?q=<検索語>` … 検索結果の棚
  - `shelf.html?ndc=<分類記号>` … その NDC 分類の棚（1〜3 桁）
  - クエリなし … 既定データ（現行の日本近代文学）の棚（後方互換・フォールバック）
- 本棚ページからスタートページへ戻る導線（ロゴ／リンク）を置く。

```
スタートページ
 ├─ 検索窓へ入力・送信 ─────────────────────► 本棚ページ ?q=<語>
 └─ NDC 分類ボタン（各階層 原則 10 個ずつ）
      1 桁（類目 0〜9）
       ├─「この分類の棚を見る」───────────► 本棚ページ ?ndc=<1桁>
       └─ 類を選択 → 2 桁（綱目 例: 0 → 00〜09）
            ├─「上の階層へ戻る」
            ├─「この分類の棚を見る」──────► 本棚ページ ?ndc=<2桁>
            └─ 綱を選択 → 3 桁（細目 例: 03 → 030〜039）
                 ├─「上の階層へ戻る」
                 └─ 目を選択 ──────────────► 本棚ページ ?ndc=<3桁>
```

> NDC（日本十進分類法）の区分は、正式には 1 桁＝類（類目表・10 区分）、
> 2 桁＝綱（綱目表・100 区分）、3 桁＝目（要目表・1,000 区分）。本書では
> 3 桁を「細目」とも呼ぶ。対象は **10 ＋ 100 ＋ 1,000 ＝ 最大 1,110 分類**。

## スタートページの要件

- **検索窓**: 語を入力して送信すると `shelf.html?q=<語>` へ遷移する
  （検索の実行自体は本棚ページ側が行う。スタートページは API を叩かない）。
- **NDC 分類ボタン**:
  - 1 桁（類目）→ 2 桁（綱目）→ 3 桁（細目）の 3 階層を、**各階層 原則 10 個**の
    ボタンで掘り下げる（例: `0〜9` → `00〜09` → `030〜039`）。
  - 各ボタンには**分類記号＋分類名**を表示する（分類名は NDC マスタから。下記）。
    データ未整備・0 件の分類はボタンを無効表示にしてよい（件数を併記できると望ましい）。
  - **「上の階層へ戻る」**: 2 桁・3 桁の表示中に 1 つ上の階層へ戻れる。
  - **「この分類の棚を見る」**: 1 桁・2 桁の段階で、それ以上絞り込まずに
    `shelf.html?ndc=<記号>` へ遷移できる。3 桁は選択＝遷移。
- **出典表示**: 分類名は日本図書館協会（JLA）公式の
  [NDC 新訂9版データ（NDC-LD・CC BY）](https://www.jla.or.jp/committees/bunrui/ndc-data/)
  を使用して表示し、スタートページ内に出典（提供元・ライセンス・リンク）を
  明記する。NDC10 版の分類名（NDC Navi 掲載相当）は転載許諾が得られた場合に
  差し替える（→「問題点と対処」#2）。
- 現行と同じく**静的 HTML＋バニラ JS・ビルド工程なし・相対パスのみ**で実装し、
  GitHub Pages（サブパス公開）でそのまま動くこと。

## トップページ（本棚ページ）の要件

- 現行の本棚 UI（タブ・並べ替え・シリーズまとめ・詳細オーバーレイ・無限スクロール）を
  そのまま引き継ぐ。**`books.json` スキーマ＝唯一の契約は不変**。
- **初期条件の受け取り**:
  - `?q=<語>` … 読み込み時に検索を実行して棚を作る（API 必須。失敗時はメッセージ＋既定棚）。
  - `?ndc=<記号>` … `site/data/ndc/<記号>.json` を読み込んで棚を作る
    （**静的ファイルのみで完結。サーバ不要**）。
  - どちらも無ければ既定データ `site/data/books.json`（現行どおり）。
- **棚見出し**: NDC 棚では分類記号・分類名・データ取得日（meta）・出典を棚上部に表示する。
- **NDC 棚内の検索**:
  - サーバ稼働時 … `/api/search?q=<語>&ndc=<記号>` で分類内を検索する
    （[server/README](../server/README.md)）。
  - サーバ未稼働時 … 読み込み済みの NDC データを**クライアント側で絞り込む**
    （server のローカル代役と同じ思想。タイトル・著者・出版社等の部分一致）。
    これにより**サーバ未稼働時も NDC ごとの表示・検索が成立**する。
- **URL 状態同期**: `q` / `ndc` に加えタブ・並べ替えもクエリへ反映し、
  共有・ブックマーク・戻る/進むに対応する（従来ロードマップの「URL 状態同期」を
  本件の前提作業としてここへ統合する）。

## データ設計

`books.json` スキーマ（[ルート README](../README.md#表示用データのスキーマ契約)）を
唯一の契約として維持したまま、**分類ごとの静的データ**を追加する。

```
site/data/
├── books.json            # 既定データ（現行どおり・クエリなし時のフォールバック）
├── meta.json
└── ndc/
    ├── index.json        # NDC マスタ: 1,110 分類の記号・分類名・件数・データ有無・出典
    ├── 9.json            # 類目（1 桁）の棚データ（books.json と同一スキーマ）
    ├── 91.json           # 綱目（2 桁）
    └── 910.json          # 細目（3 桁）
```

- **NDC マスタ（`index.json`）**: 全 1,110 分類の
  `{ code, label, count, records, hasData, fetchedAt }` と、分類名の出典情報
  （`labelSource`＝NDC-LD・取得日）・生成日時を持つ。
  スタートページのボタン描画と本棚ページの棚見出しが参照する。
- **分類ごとの棚データ（`<code>.json`）**: `books.json` と**同一スキーマ**の配列
  （`ownerCount` 降順）。**1 分類あたり最大 10,000 件程度**を想定し、上限超過分は
  所蔵館数上位を優先して切り詰める（上限値は実測後に確定。→「問題点と対処」#3）。
- **リポジトリに保存**して GitHub Pages で静的配信する。これにより
  **サーバ未稼働でも NDC ごとの表示・検索が可能**（グレースフルデグレードの拡張）。
- データは定期的に再取得して更新する（下記パイプライン）。

## 取得・更新パイプライン

```
fetch/ndc_fetch.py（新設・バッチ）
  1,110 分類を CiNii の分類検索で取得 → 生レスポンスを .cache/ndc/ へキャッシュ
        │（冪等・中断再開可能・レート制限などのマナーは fetch の既存方針を踏襲）
        ▼
build/build.py --ndc（拡張）
  core.normalize で正規化・整列（既定データと同一ロジック）
        │
        ▼
site/data/ndc/*.json ＋ index.json      → コミットして公開
```

- **取得 API**: 初回整備は現行 CiNii Books OpenSearch（1 コール最大 10,000 件）を
  想定するが、**分類検索パラメータの有無・前方一致の可否は実装前に要確認**
  （→「問題点と対処」#1）。CiNii Research 移行（P1）後は新 API
  （appid 必須・200 件/コール）へ引き継ぐため、取得層は差し替え可能に作る
  （[core/README](../core/README.md)）。
- **定期更新**: 当面は**手動実行→コミット**で運用する。GitHub Actions の
  `schedule` による自動化は CI/CD 変更のためユーザーレビューを経て別途行う。
  全分類を一度に更新せず、分割（例: 週ごとに 1/4 ずつ）してもよい。
- NDC マスタ（分類名）は分類の改訂がない限り不変なので、初回整備後は再取得不要。

## 問題点と対処（実装前の必須確認を含む）

| # | 問題 | 対処 |
|---|---|---|
| 1 | **分類検索 API の可否が未確認**。CiNii Books の詳細検索には「分類」欄があるが、OpenSearch の対応パラメータ（`clas` 等）・上位桁での前方一致の可否は公式仕様で未確認（仕様ページはデータセンターからのアクセスが 403 で検証未了） | **確認済み（2026-07-13・実レスポンス `source/0.json`）**: `clas` パラメータと上位桁の前方一致（`clas=0*`）が有効。`sortorder=5`（所蔵館数降順）・`count=10000` も分類検索で機能し、**件数超過時は API 側が所蔵館数上位 1 万件で切り詰めて返す**（totalResults 230,095 に対し items 10,000 件・ownerCount 832→132 の降順を確認）。レスポンス構造は既定データと同一（`@graph[0].items`）で core の正規化が無改修で通る。CiNii Research 移行後の同等機能は P1 移行時に確認する |
| 2 | **NDC 分類名の利用許諾**。NDC の著作権は日本図書館協会（JLA）に帰属し、綱目表等の転載には掲載許諾を得ている例がある。NDC Navi（大阪公立大学・NDC・MRDF が元）からの引用条件も未確認 | **調査済み（2026-07-14）**: ① NDC Navi は第三者向けの利用規約を公開しておらず、サイト自体が「NDC・MRDF10 を使用し、公開にあたって JLA の許可を得ています」（About ページ）という立て付けのため、**転載には JLA・開発者（大阪公立大 村上研究室）への許諾申請が別途必要**。② 一方 **JLA 自身が [NDC 新訂8版・9版データ（NDC-LD）を CC-BY で公式公開](https://www.jla.or.jp/committees/bunrui/ndc-data/)**（NDL との共同研究成果・2025-07 からサイト直接配布）。→ **分類名は NDC-LD（NDC9 版）を出典表示つきで使用する**（`fetch/ndc_labels.py` で抽出。1,110 分類中 1,050 を収録、残り 60 は NDC9 の欠番）。NDC10 版の分類名は許諾が得られた場合に差し替える（ユーザー作業） |
| 3 | **データ総量**。最悪 1,110 分類 × 10,000 件 ≈ 1,100 万レコード（既定データの実測 5,212 件 ≒ 1.9MB から換算して**約 4GB 級**）となり、GitHub リポジトリ（推奨 1GB 未満）・Pages（サイト 1GB・1 ファイル 100MB）の上限を超えうる | **先に全分類の件数だけを実測**（`fetch/ndc_fetch.py --counts`。1 分類 1 コールで総件数のみ取得）して総量を見積り、①1 分類あたりの件数上限の引き下げ（所蔵館数上位優先。`build.py --ndc-max`）②上位階層（類・綱）は細目の集約＋上位 N 件に制限 ③圧縮保存、等の組み合わせを実測値で確定する。**サイズ実測（2026-07-13・`source/0.json` 10,000 件）**: 正規化後 4.19MB（1 件平均 419 バイト）・gzip 転送 0.90MB。**全 1,110 分類の件数実測（2026-07-14・`ndc_fetch.py --counts`）**: 延べ 18,686,614 件（類 613 万／綱 624 万／目 632 万。3 桁の中央値 2,471 件・0 件の分類なし）。上限ごとの総量: 10,000 件＝約 2.05GB（不可）／2,000 件＝約 706MB／1,000 件＝約 392MB／500 件＝約 210MB。→ **上限は 1,000 件に確定（ユーザー確認済み 2026-07-14。総量約 392MB・収載約 93.7 万件）** |
| 4 | **取得コール数・所要時間**。CiNii Research API（200 件/コール）では最大約 5.5 万コール。マナー（間隔 1 秒）を守ると数日規模 | 分割実行・キャッシュ・中断再開可能な設計（fetch の既存方針）で吸収する。初回整備は現行 CiNii Books API（1 コール 10,000 件 → 全体 1,110 コール規模）のうちに行うのが現実的 |
| 5 | **Git 履歴の肥大**。定期更新のたびに大容量 JSON の差分が履歴へ積まれる | 更新頻度を抑える（月次等）。必要ならデータ専用ブランチを履歴なしで強制更新する等の方式を実装時に選定する |
| 6 | **10,000 件棚のクライアント負荷**。1 ファイル約 4MB（gzip 転送で約 700KB 想定） | 描画は既存の無限スクロール（100 件ずつ）で問題ない。初期ロードの体感が悪ければ分割ロードへ切り替える（スキーマ不変のまま出力構造で対応。→ [build/README](../build/README.md) 設計方針） |

いずれも**方針を覆すものではなく**、手順 0（事前調査・実測）の結果で
パラメータ（件数上限・保存方式・更新頻度）を確定すれば吸収できる。

## 受け入れ条件（この構成変更の完了定義）

1. `/`（スタートページ）から、検索語または NDC 分類（1〜3 桁のどこで確定してもよい）を
   指定して本棚ページへ遷移できる。
2. NDC ボタンは各階層 10 個ずつ表示され、「上の階層へ戻る」「この分類の棚を見る」
   （1・2 桁時）が機能する。
3. 分類名が JLA 公式 NDC データ（NDC9 版・CC BY）で表示され、出典表示が
   スタートページ・棚見出しにある（NDC10 版へは許諾取得後に差し替え可能）。
4. `shelf.html?ndc=<記号>` が**サーバ未稼働（GitHub Pages のみ）で**表示・棚内検索できる。
5. `shelf.html`（クエリなし）は現行どおり既定データの棚を表示する（後方互換）。
6. `site/data/ndc/` の再生成が冪等で、`index.json` から各分類の件数・取得日を確認できる。
